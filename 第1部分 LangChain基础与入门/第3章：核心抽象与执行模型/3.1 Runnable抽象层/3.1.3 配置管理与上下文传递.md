# 3.1.3 配置管理与上下文传递

## 概念讲解：为什么需要配置传递？

### 传统配置管理的痛点

```python
# ❌ 硬编码配置，无法动态修改
llm = ChatOpenAI(model="gpt-4", temperature=0.7)
result = llm.invoke("你好")  # 温度固定死了

# ❌ 每次调用都要重新创建实例
llm_low_temp = ChatOpenAI(model="gpt-4", temperature=0.1)
llm_high_temp = ChatOpenAI(model="gpt-4", temperature=0.9)
```

> **核心要点**：配置与实例分离，运行时动态传参。

## 核心要点：RunnableConfig

### RunnableConfig的作用

RunnableConfig是LangChain的**配置传递容器**，在调用时传入，不影响组件实例：

```python
from langchain_core.runnables import RunnableConfig
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

# 一次调用，临时配置
result = llm.invoke(
    "写首诗",
    config=RunnableConfig(
        tags=["creative"],           # 标签（用于追踪）
        metadata={"user": "alice"},  # 元数据（上下文信息）
        max_concurrency=5,           # 并发限制
    )
)

# 另一次调用，不同配置
result = llm.invoke(
    "写份报告",
    config=RunnableConfig(
        tags=["formal"],
        metadata={"user": "bob"},
    )
)
```

> **关键洞察**：同一个实例，每次调用可以有不同的配置。

### 配置项说明

| 配置项 | 类型 | 作用 |
|--------|------|------|
| tags | List[str] | 打标签，用于过滤和追踪 |
| metadata | Dict | 传递任意上下文信息 |
| callbacks | List[CallbackHandler] | 回调处理器列表 |
| max_concurrency | int | 最大并发数 |
| run_name | str | 运行名称（用于日志追踪） |

## 简单示例：追踪执行情况

### 场景：通过metadata传递用户信息

```python
from langchain_core.runnables import RunnableLambda, RunnableConfig

def log_request(x, config: RunnableConfig = None):
    """记录请求信息"""
    if config and config.metadata:
        print(f"用户: {config.metadata.get('user_id')}")
        print(f"环境: {config.metadata.get('env')}")
    return x

chain = RunnableLambda(log_request) | llm

# 传入用户上下文
chain.invoke(
    "你好",
    config=RunnableConfig(
        metadata={
            "user_id": "12345",
            "env": "production",
            "region": "cn-shanghai",
        }
    )
)
```

## 进阶理解：回调系统

### 什么是回调？

回调是**监听执行过程**的机制，LangChain通过callbacks提供：

```python
from langchain_core.callbacks import BaseCallbackHandler

class MyCallback(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"LLM开始: {prompts[0][:30]}...")
    
    def on_llm_end(self, response, **kwargs):
        print(f"LLM完成: {response.generations[0][0].text[:30]}...")

# 使用回调
result = llm.invoke(
    "你好",
    config=RunnableConfig(callbacks=[MyCallback()])
)
```

### 回调事件生命周期

```
on_chain_start → on_llm_start → on_llm_end → on_chain_end
```

### with_config：快捷配置覆盖

```python
# with_config创建配置覆盖的Runnable
creative_llm = llm.with_config({"tags": ["creative"]})
formal_llm = llm.with_config({"tags": ["formal"]})

# 分别调用
creative_llm.invoke("写首诗")  # 带有creative标签
formal_llm.invoke("写份报告")  # 带有formal标签
```

## 常见问题

### Q1: config和with_config有什么区别？

| 方式 | 时机 | 适用场景 |
|------|------|---------|
| invoke(config=...) | 调用时 | 临时一次性配置 |
| with_config(...) | 创建时 | 多次复用不同配置 |

```python
# invoke时传入 - 每次调用不同
llm.invoke("你好", config=RunnableConfig(tags=["test"]))

# with_config - 创建新Runnable
configured_llm = llm.with_config({"tags": ["test"]})
configured_llm.invoke("你好")  # 复用配置
```

### Q2: 如何在链中间读取配置？

```python
from langchain_core.runnables import RunnableConfig

def read_config(x, config: RunnableConfig = None):
    # LangChain自动注入config参数
    user_id = (config.metadata or {}).get("user_id", "anonymous")
    return f"处理用户 {user_id} 的请求: {x}"

chain = RunnableLambda(read_config) | llm
```

### Q3: metadata和tags有什么用？

```python
# 实际应用场景
config = RunnableConfig(
    tags=["rag", "production"],     # 用于过滤：只看RAG相关的执行
    metadata={                       # 用于追踪：关联业务上下文
        "user_id": "123",
        "document_id": "doc_456",
        "query_type": "search",
    }
)
```

## 本节总结

### 核心认知
1. **RunnableConfig**：配置传递容器，与实例分离
2. **tags/metadata**：打标签和传递上下文信息
3. **callbacks**：监听执行过程的钩子

### 关键启示
- 优先使用with_config创建可复用的配置
- metadata适合传递业务上下文
- callbacks是调试和监控的重要工具
