# 1.3.2 LangChain安装与版本控制

## 概念讲解：LangChain的包架构

### 为什么LangChain拆分成多个包？

在v0.2版本之后，LangChain采用了模块化包结构，这是理解安装策略的前提：

```
┌──────────────────────────────────────────────────────────┐
│              LangChain 模块化包结构                        │
│                                                          │
│  langchain-core                                          │
│  ├─ 核心抽象和接口                                        │
│  ├─ Runnable协议、Message类型、Prompt模板                  │
│  └─ 所有LangChain组件的基础                               │
│                                                          │
│  langchain                                               │
│  ├─ 高层组件                                              │
│  ├─ Chains、Agents、Retrievers                           │
│  └─ 依赖langchain-core                                    │
│                                                          │
│  langchain-{provider}（如langchain-openai）               │
│  ├─ 特定模型提供商的集成                                   │
│  ├─ 每个集成独立维护、独立版本                             │
│  └─ 按需安装，不依赖其他提供商                             │
│                                                          │
│  langchain-community                                     │
│  ├─ 社区贡献的集成                                        │
│  ├─ 非官方维护的第三方集成                                 │
│  └─ 质量参差不齐，需自行评估                               │
│                                                          │
│  langgraph                                               │
│  ├─ 工作流编排引擎                                        │
│  ├─ 支持复杂的状态图、条件分支、循环                       │
│  └─ 独立于langchain的核心包                               │
└──────────────────────────────────────────────────────────┘
```

> **核心要点**：模块化设计的核心优势是**按需安装、减少依赖、独立更新**。你不需要为不使用的功能安装依赖。

### 版本演进历史

了解版本历史有助于理解为什么教程基于特定版本：

```
┌─────────────────────────────────────────────────────────┐
│               LangChain版本演进                           │
│                                                         │
│  v0.1.x (2023年初)                                      │
│  ├─ 第一个稳定版本                                       │
│  ├─ 所有功能在一个包中                                   │
│  └─ API设计初步成型                                      │
│                                                         │
│  v0.2.x (2023年底)                                      │
│  ├─ 重大重构：拆分为多包架构                             │
│  ├─ langchain-core独立                                   │
│  ├─ 大量API被标记为deprecated                           │
│  └─ 引入LCEL（LangChain Expression Language）            │
│                                                         │
│  v0.3.x (2024年中)                                      │
│  ├─ API趋于稳定                                          │
│  ├─ 更多集成加入                                         │
│  ├─ LangGraph成熟                                        │
│  └─ 生产级特性增强                                       │
│                                                         │
│  v1.x (当前稳定版)                                      │
│  ├─ API正式稳定，向后兼容                                │
│  ├─ 完整的生态系统                                       │
│  └─ 企业级特性完善                                       │
└─────────────────────────────────────────────────────────┘
```

## 核心要点：安装策略

### 最小安装（推荐入门）

对于初学者，只需安装核心包：

```bash
# 使用uv（推荐）
uv add "langchain>=1.2.22" langchain-openai

# 或使用pip
pip install "langchain>=1.2.22" langchain-openai
```

**这个最小安装包含**：
- `langchain-core`：核心抽象（自动依赖）
- `langchain`：高层组件
- `langchain-openai`：OpenAI模型集成

**可以做什么**：
- 创建模型实例并调用
- 构建提示词模板
- 使用链式语法组合组件
- 基本的对话和RAG功能

### 完整安装（按需扩展）

根据项目需求添加额外组件：

```bash
# 工作流编排（复杂流程需要）
uv add langgraph

# 向量数据库（RAG应用需要）
uv add langchain-chroma chromadb

# 监控调试（生产环境推荐）
uv add langsmith

# 社区集成（特殊需求）
uv add langchain-community

# 本地模型（不想用API）
uv add langchain-ollama
```

### 项目结构模板

一个标准的LangChain项目应该包含以下结构：

```
my-langchain-project/
├── .env                    # 环境变量（不提交到Git）
├── .gitignore              # Git忽略配置
├── pyproject.toml          # 项目配置和依赖
├── uv.lock                 # 依赖锁定文件（确保可复现）
├── README.md               # 项目说明
├── src/
│   ├── __init__.py
│   ├── chains/             # 链式组件
│   ├── agents/             # 代理系统
│   ├── tools/              # 自定义工具
│   └── utils/              # 工具函数
├── tests/                  # 测试代码
└── notebooks/              # Jupyter探索笔记
```

## 简单示例：验证安装

创建一个简单的测试脚本来验证安装：

```python
# verify_install.py - 验证LangChain安装
"""
运行此脚本验证LangChain是否正确安装。
用法：uv run python verify_install.py
"""

def verify_installation():
    """验证LangChain环境是否正确"""
    
    print("=" * 50)
    print("LangChain 安装验证")
    print("=" * 50)
    
    # 1. 验证langchain-core
    try:
        import langchain_core
        print(f"✅ langchain-core: {langchain_core.__version__}")
    except ImportError as e:
        print(f"❌ langchain-core 未安装: {e}")
        return False
    
    # 2. 验证langchain
    try:
        import langchain
        print(f"✅ langchain: {langchain.__version__}")
    except ImportError as e:
        print(f"❌ langchain 未安装: {e}")
        return False
    
    # 3. 验证langchain-openai
    try:
        from langchain_openai import ChatOpenAI
        print("✅ langchain-openai: 已安装")
    except ImportError as e:
        print(f"⚠️  langchain-openai 未安装: {e}")
        print("   如使用OpenAI模型，请安装: uv add langchain-openai")
    
    # 4. 验证Runnable协议
    try:
        from langchain_core.runnables import Runnable
        print("✅ Runnable协议: 可用")
    except ImportError as e:
        print(f"❌ Runnable协议 不可用: {e}")
        return False
    
    print("\n" + "=" * 50)
    print("✅ 所有核心组件验证通过！")
    print("   可以开始LangChain开发了。")
    print("=" * 50)
    return True

if __name__ == "__main__":
    verify_installation()
```

运行验证：
```bash
uv run python verify_install.py
```

## 进阶理解：版本控制策略

### 语义化版本理解

LangChain遵循语义化版本（Semantic Versioning）：

```
主版本.次版本.修订版本
  ↑      ↑      ↑
  |      |      └─ 向后兼容的Bug修复
  |      └─ 向后兼容的新功能
  └─ 不兼容的API变更
```

**版本兼容性指南**：

| 版本变化 | 兼容性 | 升级建议 |
|---------|--------|---------|
| 修订版本 (1.2.22 → 1.2.23) | 完全兼容 | 直接升级 |
| 次版本 (1.2 → 1.3) | 向后兼容 | 测试后升级 |
| 主版本 (1.x → 2.x) | 可能不兼容 | 仔细阅读迁移指南 |

### 依赖版本约束

```toml
# pyproject.toml中的版本约束语法

# 精确版本
dependencies = ["langchain==1.2.22"]

# 最小版本（推荐）
dependencies = ["langchain>=1.2.22"]

# 版本范围
dependencies = ["langchain>=1.2.22,<2.0.0"]

# 兼容版本（~等价于>=1.2.22,<1.3.0）
dependencies = ["langchain~=1.2.22"]
```

**推荐策略**：使用`>=最小版本`配合锁文件，既保证能获取Bug修复，又通过锁文件确保一致性。

### 处理版本冲突

遇到依赖冲突时的排查步骤：

```bash
# 1. 查看依赖树
uv pip tree

# 2. 查看特定包的依赖
uv pip show langchain

# 3. 检查冲突
uv pip check

# 4. 强制重新解析依赖
uv sync --upgrade
```

**常见冲突场景**：

| 冲突类型 | 原因 | 解决方案 |
|---------|------|---------|
| pydantic版本冲突 | 不同包要求不同pydantic版本 | 升级到兼容版本或使用uv自动解决 |
| numpy版本冲突 | 科学计算包之间的numpy要求 | 指定统一的numpy版本 |
| Python版本不兼容 | 包要求更高Python版本 | 升级Python或降级包版本 |

## 常见问题

### Q1: 应该使用哪个版本的LangChain？
**A:** 
- **学习/新项目**：使用最新的稳定版（当前为1.x）
- **维护旧项目**：保持当前版本，不要随意升级
- **生产环境**：固定版本号，经过充分测试后再部署

### Q2: v0.x和v1.x的区别大吗？
**A:** 非常大。v1.x进行了重大重构：
- 包结构拆分（langchain-core独立）
- API重构（引入LCEL表达式语言）
- 许多v0.x的API被标记为deprecated
- 新增LangGraph集成

**强烈建议使用v1.x**，v0.x已不再维护。

### Q3: 如何查看当前安装的LangChain版本？
**A:** 多种方式：

```python
# Python中查看
import langchain
print(langchain.__version__)

# 命令行查看
uv pip show langchain
# 或
pip show langchain
```

### Q4: 是否需要安装langchain-community？
**A:** 按需安装。langchain-community包含大量社区贡献的集成：
- **需要时安装**：使用特定社区集成（如某些数据库、工具）
- **不需要时不装**：只用官方集成（OpenAI、Anthropic等）时不需要
- **注意**：社区集成质量参差不齐，生产环境需自行评估

## 本节总结

### 核心要点
1. **模块化安装**：按需安装，最小安装即可开始开发
2. **版本固定**：使用pyproject.toml和锁文件确保可复现性
3. **验证安装**：通过脚本验证环境是否正确

### 关键启示
- 理解包结构有助于选择合适的组件
- 版本约束配合锁文件是最佳实践
- 遇到冲突时善用工具排查
