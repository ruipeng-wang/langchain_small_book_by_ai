# 1.2.2 LangGraph工作流编排

## 概念讲解：从线性链到图计算

### 为什么需要LangGraph？

LangChain的基础链式结构擅长处理**线性流程**，但实际业务场景往往更加复杂：

```
┌──────────────────────────────────────────────────────┐
│              LangChain Chain的局限性                   │
│                                                      │
│  链式结构：A → B → C → D                             │
│                                                      │
│  实际业务需要的：                                     │
│  • 条件分支：如果A的结果是X，走B；如果是Y，走C         │
│  • 循环迭代：重复执行直到满足条件                       │
│  • 并行处理：B和C同时执行，然后合并结果                 │
│  • 状态保持：在多轮执行中维护上下文状态                 │
│  • 人工介入：在关键节点等待人工审核                     │
└──────────────────────────────────────────────────────┘
```

> **核心要点**：LangChain的Chain适合"流水线"，LangGraph适合"交通网络"。前者是线性的，后者是有向图，可以表达任意复杂的业务流程。

### 什么是图（Graph）？

在计算机科学中，**图**是一种由节点（Node）和边（Edge）组成的数据结构：

```
┌────────────────────────────────────────────────────────┐
│                    图的基本概念                          │
│                                                        │
│     [节点A] ─────→ [节点B]                              │
│        │              │                                │
│        │              ↓                                │
│        └───────→ [节点C] ←── [节点D]                    │
│                                                        │
│  节点（Node）：执行单元（函数、组件、子图）               │
│  边（Edge）：执行路径，定义节点间的连接关系               │
│  有向图：边有方向，A→B不等于B→A                          │
│  循环图：允许回到之前的节点（实现循环）                   │
└────────────────────────────────────────────────────────┘
```

### LangGraph的设计哲学

LangGraph将AI工作流建模为**状态图（State Graph）**：

```python
# 核心概念：状态图 = 状态定义 + 节点函数 + 边规则

from langgraph.graph import StateGraph

# 1. 定义状态：在整个工作流中传递的数据
class MyState(TypedDict):
    query: str          # 用户输入
    result: str         # 最终结果
    steps: list         # 执行步骤记录

# 2. 定义节点：每个节点是一个执行单元
def step_a(state):
    return {"steps": state["steps"] + ["执行了A"]}

def step_b(state):
    return {"steps": state["steps"] + ["执行了B"]}

# 3. 构建图：定义节点间的连接关系
graph = StateGraph(MyState)
graph.add_node("A", step_a)
graph.add_node("B", step_b)
graph.add_edge("A", "B")  # A执行完后执行B
```

## 核心要点：LangGraph的核心特性

### 特性一：条件路由（Conditional Edges）

根据当前状态动态决定下一步执行哪个节点：

```python
from langgraph.graph import END

def route_by_intent(state):
    """根据用户意图路由到不同处理流程"""
    if "天气" in state["query"]:
        return "weather_node"
    elif "计算" in state["query"]:
        return "calculator_node"
    else:
        return "chat_node"

# 添加条件边：根据返回值决定下一个节点
graph.add_conditional_edges(
    "router",           # 从哪个节点出发
    route_by_intent,    # 路由函数
    {                   # 路由映射
        "weather_node": "weather",
        "calculator_node": "calculator",
        "chat_node": "chat"
    }
)
```

### 特性二：循环执行（Cyclic Graph）

允许图中存在循环，实现迭代和重试逻辑：

```
┌──────────────────────────────────────┐
│          循环执行场景示例              │
│                                      │
│  [Agent思考] → [执行工具] → [检查结果] │
│      ↑                              │
│      └───── 结果不满意，继续循环       │
│                                      │
│  典型应用：                           │
│  • Agent的ReAct循环                   │
│  • 代码生成的测试-修复循环             │
│  • 多轮对话的上下文更新               │
└──────────────────────────────────────┘
```

### 特性三：状态持久化（Checkpointing）

支持保存和恢复执行状态，实现断点续跑：

```python
from langgraph.checkpoint.memory import MemorySaver

# 创建检查点存储器
checkpointer = MemorySaver()

# 编译图时启用检查点
app = graph.compile(checkpointer=checkpointer)

# 执行时可以从中断点恢复
config = {"configurable": {"thread_id": "conversation_123"}}
result = app.invoke({"query": "你好"}, config)

# 即使进程重启，也可以从上次中断的地方继续
```

### 特性四：子图嵌套（Subgraphs）

支持将复杂图拆分为多个子图，提高可维护性：

```python
# 创建一个独立的RAG子图
rag_graph = StateGraph(RAGState)
rag_graph.add_node("retrieve", retrieve_documents)
rag_graph.add_node("generate", generate_answer)
rag_graph.add_edge("retrieve", "generate")
rag_subgraph = rag_graph.compile()

# 在主图中使用子图
main_graph = StateGraph(MainState)
main_graph.add_node("rag", rag_subgraph)  # 子图作为一个节点
main_graph.add_node("router", route_query)
main_graph.add_edge("router", "rag")
```

## 简单示例：构建一个简单的条件路由图

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

# 1. 定义状态
class ChatState(TypedDict):
    message: str
    response: str

# 2. 定义节点函数
def classify_intent(state: ChatState) -> dict:
    """简单分类：判断是问候还是问题"""
    greeting_words = ["你好", "hello", "hi", "早上好", "晚上好"]
    is_greeting = any(word in state["message"].lower() for word in greeting_words)
    return {"intent": "greeting" if is_greeting else "question"}

def handle_greeting(state: ChatState) -> dict:
    """处理问候"""
    return {"response": f"你好！很高兴见到你。有什么我可以帮助你的吗？"}

def handle_question(state: ChatState) -> dict:
    """处理问题（这里简化为固定回复）"""
    return {"response": f"关于'{state['message']}'，这是一个很好的问题！让我想想..."}

# 3. 构建图
graph = StateGraph(ChatState)

# 添加节点
graph.add_node("classify", classify_intent)
graph.add_node("greeting", handle_greeting)
graph.add_node("question", handle_question)

# 添加边
graph.add_edge(START, "classify")  # 从入口到分类

# 添加条件边：根据分类结果路由
graph.add_conditional_edges(
    "classify",
    lambda s: s.get("intent", "question"),
    {"greeting": "greeting", "question": "question"}
)

# 两个处理节点都连接到终点
graph.add_edge("greeting", END)
graph.add_edge("question", END)

# 4. 编译并执行
app = graph.compile()
result = app.invoke({"message": "你好"})
print(result["response"])  # 输出问候回复
```

## 进阶理解：LangGraph在Agent中的应用

### Agent本质上是一个LangGraph

LangChain的Agent系统底层就是基于LangGraph实现的：

```
┌─────────────────────────────────────────────────┐
│             Agent的图结构表示                     │
│                                                 │
│  [用户输入] → [LLM思考] ──→ [有工具调用？]       │
│                   ↑              │               │
│                   │         No   │   Yes         │
│                   │              ↓               │
│                   │         [执行工具]            │
│                   │              │               │
│                   │         [结果反馈] ──────────┘
│                   │
│              [输出回复]
│                   ↓
│               [用户]
└─────────────────────────────────────────────────┘
```

### ReAct模式的图实现

```python
# ReAct = Reasoning + Acting
# 思考→行动→观察→思考→行动...直到得出答案

def agent_loop(state):
    """Agent思考节点"""
    messages = state["messages"]
    response = llm_with_tools.invoke(messages)
    return {"messages": [response]}

def tool_loop(state):
    """工具执行节点"""
    last_message = state["messages"][-1]
    tool_name = last_message.tool_calls[0]["name"]
    tool_input = last_message.tool_calls[0]["args"]
    result = tools[tool_name].invoke(tool_input)
    return {"messages": [ToolMessage(content=result)]}

def should_continue(state):
    """判断是否继续循环"""
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tools"  # 需要调用工具，继续循环
    return END  # 没有工具调用，结束

# 构建ReAct图
react_graph = StateGraph(AgentState)
react_graph.add_node("agent", agent_loop)
react_graph.add_node("tools", tool_loop)
react_graph.add_edge(START, "agent")
react_graph.add_conditional_edges("agent", should_continue)
react_graph.add_edge("tools", "agent")  # 工具执行后回到Agent

react_app = react_graph.compile()
```

## 常见问题

### Q1: 什么时候用Chain，什么时候用LangGraph？
**A:** 简单判断标准：
- **用Chain**：线性流程，A→B→C，没有分支和循环
- **用LangGraph**：需要条件分支、循环、状态保持、人工介入

### Q2: LangGraph的学习难度大吗？
**A:** 基础使用很简单，掌握状态定义和节点函数即可。高级功能（子图、检查点、流式执行）可以按需学习。建议从简单示例开始，逐步增加复杂度。

### Q3: LangGraph是否替代了LangChain？
**A:** 不是替代关系，而是互补关系。LangGraph专注于工作流编排，而LangChain提供模型集成、提示模板等基础组件。实际应用中，两者通常一起使用。

## 本节总结

### 核心认知
1. **从线性到图结构**：LangGraph突破了Chain的线性限制，支持任意复杂的业务流程
2. **状态是核心**：所有节点通过状态传递数据，状态定义了图的"记忆"
3. **Agent的底层实现**：理解LangGraph有助于深入理解Agent的工作原理

### 关键启示
- 简单应用用Chain，复杂流程用LangGraph
- 条件路由和循环是LangGraph最强大的两个特性
- 检查点机制让生产环境的可靠执行成为可能
