# 2.2.3 错误处理与重试机制

## 概念讲解：为什么AI应用需要特殊的错误处理？

### 传统错误处理 vs AI错误处理

```
❌ 传统应用的错误：
   - 数据库连接失败 → 明确的异常类型
   - 文件不存在 → 明确的错误码
   - 网络超时 → 可预测的超时异常
   
✅ AI应用的额外错误：
   - LLM返回格式不符合预期 → 解析失败
   - API限流（Rate Limit）→ 429错误
   - 模型临时不可用 → 503错误
   - Token超出限制 → 400错误
   - 返回内容质量差 → 需要重试
```

> **核心要点**：AI应用的错误处理需要**自动重试、优雅降级、 fallback回退**等机制。

## 核心要点：LangChain的重试机制

### with_retry：内置重试包装器

LangChain为所有Runnable组件提供`with_retry`方法：

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

# 添加重试：最多3次，遇到特定异常时重试
retry_llm = llm.with_retry(
    stop_after_attempt=3,        # 最多重试3次
    wait_exponential_jitter=True # 指数退避+随机抖动
)
```

**重试策略**：

```
第1次调用失败
    ↓
等待 1秒 + 随机抖动
    ↓
第2次调用失败
    ↓
等待 2秒 + 随机抖动
    ↓
第3次调用失败
    ↓
等待 4秒 + 随机抖动
    ↓
第4次调用失败 → 抛出最终异常
```

### 简单示例：带重试的LLM调用

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")
prompt = ChatPromptTemplate.from_template("请回答：{question}")

# 链式添加重试
chain = prompt | llm.with_retry(stop_after_attempt=3)

try:
    result = chain.invoke({"question": "什么是机器学习？"})
    print(result.content)
except Exception as e:
    print(f"重试3次后仍然失败: {e}")
```

## 核心要点：Fallback回退机制

### with_fallbacks：优雅降级

当主组件失败时，自动切换到备用组件：

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

# 主模型和备用模型
primary_llm = ChatOpenAI(model="gpt-4o-mini")
backup_llm = ChatAnthropic(model="claude-3-haiku")

# 主模型失败时自动切换到备用模型
llm_with_fallback = primary_llm.with_fallbacks([backup_llm])
```

**执行流程**：

```
调用 primary_llm
    ↓
失败？
    ↓
是 → 调用 backup_llm
    ↓
返回backup_llm的结果
```

### 多层Fallback

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic
from langchain_google_genai import ChatGoogleGenerativeAI

# 多级备用模型
llm = (
    ChatOpenAI(model="gpt-4o")  # 主模型（最好但最贵）
    .with_fallbacks([
        ChatOpenAI(model="gpt-4o-mini"),  # 第一备用
        ChatAnthropic(model="claude-3-haiku"),  # 第二备用
        ChatGoogleGenerativeAI(model="gemini-pro")  # 第三备用
    ])
)
```

## 简单示例：完整的错误处理工作流

### 场景：健壮的RAG问答系统

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableLambda

# 1. 定义组件（带重试）
llm = ChatOpenAI(model="gpt-4o-mini").with_retry(stop_after_attempt=3)

retriever = vector_store.as_retriever().with_retry(
    stop_after_attempt=2,
    retry_if_exception_type=(ConnectionError, TimeoutError)
)

# 2. 带Fallback的LLM
robust_llm = ChatOpenAI(model="gpt-4o-mini").with_fallbacks([
    ChatOpenAI(model="gpt-3.5-turbo"),
    ChatAnthropic(model="claude-3-haiku")
])

# 3. 错误处理包装器
def safe_execute(chain, input_data):
    """安全执行链，捕获并处理异常"""
    try:
        return chain.invoke(input_data)
    except Exception as e:
        # 记录错误日志
        print(f"执行失败: {e}")
        # 返回友好的错误信息
        return {"error": True, "message": "系统暂时无法处理您的请求，请稍后重试。"}

# 4. 构建健壮的RAG链
prompt = ChatPromptTemplate.from_template("""
根据以下上下文回答问题：
{context}

问题：{question}
""")

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {
        "context": retriever | RunnableLambda(format_docs),
        "question": RunnableLambda(lambda x: x)
    }
    | prompt
    | robust_llm
    | StrOutputParser()
)

# 5. 使用
result = safe_execute(rag_chain, "LangChain的主要优势是什么？")
if isinstance(result, dict) and result.get("error"):
    print(f"错误: {result['message']}")
else:
    print(f"回答: {result}")
```

## 进阶应用：自定义错误处理策略

### 场景：智能重试判断

```python
from langchain_core.runnables import RunnableLambda

def smart_retry_handler(func):
    """智能重试：根据错误类型决定重试策略"""
    def wrapper(*args, **kwargs):
        max_retries = 3
        last_error = None
        
        for attempt in range(max_retries):
            try:
                return func(*args, **kwargs)
            except Exception as e:
                last_error = e
                
                # 判断错误类型
                error_msg = str(e).lower()
                
                # 429限流：等待更长时间后重试
                if "429" in error_msg or "rate limit" in error_msg:
                    import time
                    wait_time = 2 ** attempt + 1
                    print(f"API限流，等待{wait_time}秒后重试...")
                    time.sleep(wait_time)
                    continue
                
                # Token超限：不重试，直接抛出
                elif "token" in error_msg and "limit" in error_msg:
                    raise ValueError("输入内容过长，请缩短后重试")
                
                # 其他错误：正常重试
                else:
                    import time
                    time.sleep(1)
                    continue
        
        raise last_error
    
    return wrapper

# 使用智能重试
@smart_retry_handler
def call_llm(prompt):
    return llm.invoke(prompt)
```

### 场景：降级策略

```python
def degrade_gracefully(chain, input_data):
    """优雅降级：主流程失败时提供备用方案"""
    try:
        # 尝试正常执行
        return chain.invoke(input_data)
    except Exception as e:
        print(f"主流程失败: {e}，尝试降级方案...")
        
        try:
            # 降级方案1：使用更简单的模型
            simple_chain = (
                RunnableLambda(lambda x: f"简化回答关于'{x}'的问题")
            )
            return simple_chain.invoke(input_data)
        except Exception:
            # 降级方案2：返回默认回复
            return "抱歉，系统暂时不可用，请稍后再试。"
```

## 常见问题

### Q1: retry和fallback有什么区别？

**A:** 
- **retry**：同一个组件重试多次（解决临时性错误）
- **fallback**：切换到不同的组件（解决组件不可用）

```
retry:  A → 失败 → A → 失败 → A → 成功
fallback: A → 失败 → B → 成功
```

### Q2: 什么错误应该重试，什么不应该？

**A:** 
| 错误类型 | 是否重试 | 原因 |
|---------|---------|------|
| 网络超时 | ✅ 重试 | 临时性问题 |
| API限流(429) | ✅ 重试（等待） | 限流是暂时的 |
| 服务器错误(5xx) | ✅ 重试 | 服务端可能恢复 |
| Token超限(400) | ❌ 不重试 | 需要修改输入 |
| 认证失败(401) | ❌ 不重试 | 需要修复配置 |
| 内容违规(400) | ❌ 不重试 | 需要修改内容 |

### Q3: 如何在链中捕获特定步骤的错误？

**A:** 用RunnableLambda包装：

```python
def safe_retriever(x):
    try:
        return retriever.invoke(x)
    except Exception as e:
        print(f"检索失败: {e}")
        return []  # 返回空结果继续执行

chain = RunnableLambda(safe_retriever) | prompt | llm
```

## 本节总结

### 核心认知
1. **with_retry**：为组件添加自动重试能力
2. **with_fallbacks**：主组件失败时切换到备用
3. **错误分类**：区分临时性错误和永久性错误

### 关键启示
- AI应用的错误处理比传统应用更复杂
- 重试解决临时性问题，Fallback解决组件不可用
- 优雅降级比直接崩溃更好
