# 4.5 文档分块策略

## 概念讲解

文档分块（Text Splitting）是RAG系统中的**关键环节**。大语言模型有上下文窗口限制，且检索精度与文档块大小密切相关。合理的分块策略直接影响检索效果和回答质量。

### 为什么需要分块？

```
未分块问题:
完整文档(50000字) → 向量化 → 检索匹配度低 → LLM处理慢

分块优势:
文档块1(500字) → 向量化 → 精准匹配
文档块2(500字) → 向量化 → 精准匹配
文档块3(500字) → 向量化 → 精准匹配
```

### 分块的两个核心参数

| 参数 | 说明 | 影响 |
|------|------|------|
| **chunk_size** | 每个块的大小 | 过大降低精度，过小丢失上下文 |
| **chunk_overlap** | 块之间的重叠 | 保证上下文连贯性，但增加存储 |

### Token与字符的关系

Embedding模型通常以token为单位处理文本。不同语言的token换算不同：
- **英文**：1 token ≈ 4 字符 ≈ 0.75 单词
- **中文**：1 token ≈ 1.5-2 字符

了解这个关系有助于合理设置`chunk_size`。

## 核心要点

### 🔑 LangChain提供的分割器

| 分割器 | 特点 | 适用场景 |
|--------|------|---------|
| `RecursiveCharacterTextSplitter` | 递归分割，保留语义完整性 | **通用推荐** |
| `CharacterTextSplitter` | 按固定字符分割 | 简单场景 |
| `TokenTextSplitter` | 按token数量分割 | 精确控制token数 |
| `MarkdownHeaderTextSplitter` | 按Markdown标题分割 | Markdown文档 |
| `SemanticChunker` | 按语义相似度分割 | 高级场景 |

### 🔑 分块最佳实践

1. **chunk_size建议**：500-1000字符（根据检索精度调整）
2. **chunk_overlap建议**：chunk_size的10%-20%
3. **分割顺序**：从大分隔符到小分隔符递归分割
4. **保留元数据**：确保每个chunk继承原文档元数据

## 简单示例

### RecursiveCharacterTextSplitter（推荐）

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text = """
LangChain是一个用于构建大语言模型应用的框架。
它提供了组件化的设计模式，包括提示模板、文档加载器、
文本分割器、向量存储等核心组件。通过这些组件的组合，
开发者可以快速构建RAG应用、智能客服、知识库系统等。
"""

# 创建分割器
splitter = RecursiveCharacterTextSplitter(
    chunk_size=100,        # 每个块的最大字符数
    chunk_overlap=20,      # 块之间的重叠字符数
    separators=["\n\n", "\n", "。", "，", " ", ""]  # 分割符优先级
)

# 执行分割
chunks = splitter.split_text(text)

for i, chunk in enumerate(chunks):
    print(f"块 {i+1}: {chunk}")
    print("-" * 30)
```

### 文档分割（保留元数据）

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_core.documents import Document

# 创建带元数据的文档
doc = Document(
    page_content="这是文档内容，" * 50,
    metadata={"source": "example.txt", "page": 1}
)

# 使用split_documents方法
splitter = RecursiveCharacterTextSplitter(
    chunk_size=100,
    chunk_overlap=20,
    add_start_index=True  # 添加起始索引到元数据
)

chunks = splitter.split_documents([doc])

# 每个chunk都保留了原始元数据
for chunk in chunks:
    print(f"来源: {chunk.metadata['source']}, 内容长度: {len(chunk.page_content)}")
```

## 进阶应用

### Markdown按标题分割

Markdown文档按标题层级分割，可以保持主题完整性：

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

markdown_text = """
# 主标题1

这是第一部分的内容。

## 子标题1.1

这是子标题1.1的内容。

# 主标题2

这是第二部分的内容。
"""

# 定义要分割的标题层级
headers_to_split_on = [
    ("#", "Header1"),
    ("##", "Header2"),
]

splitter = MarkdownHeaderTextSplitter(headers_to_split_on)
chunks = splitter.split_text(markdown_text)

for chunk in chunks:
    print(f"标题: {chunk.metadata}")
    print(f"内容: {chunk.page_content[:50]}...")
    print("-" * 30)
```

### 代码文档分割

代码有特殊的结构（函数、类、模块），需要专用分割器：

```python
from langchain_text_splitters import Language, RecursiveCharacterTextSplitter

# Python代码分割器
python_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=500,
    chunk_overlap=50
)

# 支持的编程语言
# Language.PYTHON, Language.JS, Language.TS, Language.JAVA
# Language.GO, Language.RUST, Language.CPP 等

code = """
def hello():
    print("Hello")

def world():
    print("World")
"""

chunks = python_splitter.create_documents([code])
```

### 语义分块（高级）

语义分块根据内容相似度自动分割，适合长文档：

```python
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

# 语义分块根据内容相似度自动分割
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

splitter = SemanticChunker(
    embeddings=embeddings,
    breakpoint_threshold_type="percentile",  # 阈值类型
    breakpoint_threshold_amount=90           # 阈值百分比
)

text = "较长的文档内容，包含多个语义段落..."
chunks = splitter.create_documents([text])
```

## 分块策略选择指南

| 文档类型 | 推荐分割器 | chunk_size | 说明 |
|---------|-----------|------------|------|
| 通用文本 | `RecursiveCharacterTextSplitter` | 500-1000 | 默认首选 |
| Markdown | `MarkdownHeaderTextSplitter` | 按标题 | 保持主题完整 |
| 代码 | `RecursiveCharacterTextSplitter.from_language()` | 500-1000 | 按函数/类分割 |
| 长文档 | `SemanticChunker` | 自适应 | 按语义分割 |
| 短文本 | `CharacterTextSplitter` | 200-500 | 简单场景 |

## 常见问题

### Q1: 如何选择合适的chunk_size？

**A:** 根据应用场景调整：
- **问答场景**：200-500字符，提高检索精度
- **摘要场景**：1000-2000字符，保留更多上下文
- **代码场景**：500-1000字符，按函数/类分割

### Q2: 分割后的chunk如何保证语义完整性？

**A:** `RecursiveCharacterTextSplitter`按分隔符优先级递归分割，尽量在段落、句子边界处切断。合理设置`separators`参数可以避免在单词或句子中间切断。

### Q3: `split_text`和`split_documents`有什么区别？

**A:** 
- `split_text`：接收字符串，返回字符串列表
- `split_documents`：接收Document列表，返回Document列表，自动继承元数据

### Q4: 如何评估分块效果？

**A:** 可以通过以下方式评估：
1. 检查每个chunk的大小分布是否均匀
2. 人工检查chunk边界是否合理
3. 在RAG系统中测试检索精度

## 本节总结

文档分块是连接原始文档和向量检索的**关键桥梁**。合理的分块策略能显著提升RAG系统的检索精度和回答质量。

核心要点：
1. 推荐使用`RecursiveCharacterTextSplitter`作为通用分割器
2. chunk_size建议500-1000字符，overlap为10%-20%
3. 分割时保留元数据，便于后续检索过滤
4. 根据数据类型选择专用分割器（Markdown、代码等）
