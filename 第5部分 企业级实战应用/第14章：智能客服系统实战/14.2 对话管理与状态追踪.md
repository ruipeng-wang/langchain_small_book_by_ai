# 14.2 对话管理与状态追踪

## 概念讲解

多轮对话管理是智能客服系统的核心能力。用户的问题往往不是一次性能说清楚的，通常会有追问、澄清、补充信息等情况。对话管理的任务就是在连续交互中保持上下文的连贯性，让系统"记住"之前说过什么，理解当前对话处于什么阶段。

### 对话管理的核心挑战

```
上下文遗忘 → 话题切换 → 信息冗余 → 状态丢失
├── 长对话记不住    ├── 用户突然换话题   ├── 重复问同样问题  ├── 系统重启后失忆
├── 早期信息失效    ├── 多任务穿插       ├── Token超限       ├── 断线无法恢复
└── 实体信息丢失    └── 意图连续变化     └── 响应变慢        └── 体验不一致
```

### LangGraph 的解决方案

LangGraph 通过以下机制解决对话管理问题：

1. **MessagesState**：自动管理消息历史，维护完整的对话上下文
2. **Checkpointer**：持久化对话状态，支持会话恢复和断点续传
3. **SummarizationNode**：自动压缩长对话，在保持上下文的同时控制 Token 消耗
4. **add_messages**：智能合并新旧消息，避免重复和冲突

## 核心要点

### 1. 记忆组件体系

| 组件 | 功能 | 适用场景 |
|------|------|----------|
| `InMemorySaver` | 内存中保存对话状态 | 开发测试、短期会话 |
| `PostgresSaver` | PostgreSQL 持久化存储 | 生产环境、跨会话记忆 |
| `SummarizationNode` | 自动摘要压缩长对话 | 超长对话、Token 控制 |

### 2. 对话状态管理策略

**短期记忆**：当前对话轮次的完整消息，使用 `MessagesState` 的 `messages` 字段自动管理。

**中期记忆**：会话级别的上下文，通过 Checkpointer 持久化，使用 `thread_id` 区分不同会话。

**长期记忆**：跨会话的用户偏好和历史信息，需要额外的业务逻辑实现（如用户画像存储）。

### 3. 上下文压缩技术

当对话过长时，需要进行压缩以避免超出 Token 限制：

```
原始对话                        压缩后
┌─────────────────────┐        ┌─────────────────────┐
│ 用户: 我想查订单     │        │ [摘要] 用户查询订单  │
│ 助手: 订单号是多少？ │   →    │ 订单号: ORD12345     │
│ 用户: ORD12345      │        │ 最新: 预计明天送达    │
│ 助手: 预计明天送达   │        └─────────────────────┘
│ 用户: 那退货呢？     │
│ ... (继续对话)       │
└─────────────────────┘
```

## 简单示例

以下示例展示如何使用 LangGraph 构建具有完整对话记忆的多轮对话系统：

```python
from langgraph.graph import StateGraph, MessagesState, START
from langgraph.checkpoint.memory import InMemorySaver
from langchain.chat_models import init_chat_model

model = init_chat_model("gpt-4o-mini")

# 定义扩展状态
class DialogState(MessagesState):
    current_topic: str        # 当前话题
    turn_count: int           # 对话轮次
    extracted_entities: dict  # 提取的实体

def chatbot(state: DialogState):
    """核心对话节点，自动获得完整对话历史"""
    # state["messages"] 包含所有历史消息
    response = model.invoke(state["messages"])
    return {"messages": [response]}

def track_topic(state: DialogState):
    """追踪当前话题"""
    last_msg = state["messages"][-1].content
    # 简化的话题追踪
    if any(kw in last_msg for kw in ["订单", "物流", "配送"]):
        topic = "order"
    elif any(kw in last_msg for kw in ["退货", "退款", "退换"]):
        topic = "refund"
    else:
        topic = "general"
    return {"current_topic": topic}

def update_turn_count(state: DialogState):
    """更新对话轮次"""
    return {"turn_count": state.get("turn_count", 0) + 1}

# 构建状态图
builder = StateGraph(DialogState)
builder.add_node("track", track_topic)
builder.add_node("chat", chatbot)
builder.add_node("count", update_turn_count)

builder.add_edge(START, "track")
builder.add_edge("track", "chat")
builder.add_edge("chat", "count")

# 添加检查点保存器
checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 多轮对话演示
config = {"configurable": {"thread_id": "session_1"}}

messages = [
    "你好，我想查询订单",
    "订单号是ORD20231234",
    "这个订单什么时候能到？",
    "好的，那退货政策是什么？"
]

for msg in messages:
    result = graph.invoke(
        {"messages": [{"role": "user", "content": msg}]},
        config
    )
    print(f"用户: {msg}")
    print(f"助手: {result['messages'][-1].content[:50]}...")
    print(f"话题: {result.get('current_topic', 'N/A')}")
    print(f"轮次: {result.get('turn_count', 0)}")
    print("-" * 40)
```

### 带摘要的长对话管理

当对话很长时，使用 `SummarizationNode` 自动压缩历史：

```python
from langgraph.graph import StateGraph, MessagesState, START
from langgraph.checkpoint.memory import InMemorySaver
from langchain.chat_models import init_chat_model
from langchain_core.messages.utils import count_tokens_approximately
from langmem.short_term import SummarizationNode, RunningSummary

model = init_chat_model("gpt-4o-mini")

class State(MessagesState):
    context: dict[str, RunningSummary]  # 存储摘要信息

summarization_node = SummarizationNode(
    token_counter=count_tokens_approximately,
    model=model,
    max_tokens=512,            # 最大Token数
    max_tokens_before_summary=256,  # 达到此值后开始摘要
    max_summary_tokens=128,    # 摘要最大Token
)

def call_model(state):
    response = model.invoke(state["summarized_messages"])
    return {"messages": [response]}

builder = StateGraph(State)
builder.add_node("summarize", summarization_node)
builder.add_node("call_model", call_model)
builder.add_edge(START, "summarize")
builder.add_edge("summarize", "call_model")

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)
```

## 进阶应用

### 1. 实体链式记忆

在客服场景中，用户提到的实体信息（订单号、产品名、日期等）需要被持续追踪：

```python
def extract_and_remember(state: DialogState):
    """提取并记忆关键实体"""
    import re
    text = state["messages"][-1].content
    entities = state.get("extracted_entities", {})
    
    # 提取订单号
    order_match = re.search(r"订单[号]?\s*[:：]?\s*(\w+)", text)
    if order_match:
        entities["order_id"] = order_match.group(1)
    
    # 提取日期
    date_match = re.search(r"(\d{4}年?\d{1,2}月?\d{1,2}日?)", text)
    if date_match:
        entities["date"] = date_match.group(1)
    
    return {"extracted_entities": entities}
```

### 2. 对话状态恢复

使用 PostgreSQL Checkpointer 实现生产级别状态持久化：

```python
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

async def create_graph():
    db_uri = "postgresql://user:pass@localhost:5432/dbname"
    async with AsyncPostgresSaver.from_conn_string(db_uri) as checkpointer:
        await checkpointer.setup()
        
        builder = StateGraph(DialogState)
        # ... 添加节点和边 ...
        graph = builder.compile(checkpointer=checkpointer)
        return graph

# 恢复之前的对话
config = {"configurable": {"thread_id": "session_1"}}
result = await graph.ainvoke(
    {"messages": [{"role": "user", "content": "我刚才问的订单状态呢？"}]},
    config
)
# 系统能够访问到之前的对话历史
```

### 3. 情感分析与满意度追踪

通过持续分析用户消息的情感倾向，评估服务质量：

```python
def analyze_sentiment(state: DialogState):
    """分析用户情感变化"""
    messages = state["messages"]
    user_msgs = [m for m in messages if m.type == "human"]
    
    # 简化的情感分析（实际应使用专用模型）
    positive = ["谢谢", "好的", "满意", "不错", "感谢"]
    negative = ["投诉", "不满意", "差评", "问题", "错误"]
    
    score = 0
    for msg in user_msgs:
        text = msg.content.lower()
        score += sum(1 for w in positive if w in text)
        score -= sum(1 for w in negative if w in text)
    
    return {"satisfaction": max(0, min(5, 3 + score))}
```

## 常见问题

### Q1: 对话过长导致 Token 超限怎么办？

**A**: 有三种策略：
1. 使用 `ConversationBufferWindowMemory` 只保留最近 N 轮对话
2. 使用 `SummarizationNode` 自动压缩历史对话
3. 在业务层面定期"归档"对话，只保留摘要

### Q2: 如何区分不同的用户会话？

**A**: 通过 `config` 中的 `thread_id` 区分。每个用户或每次对话使用不同的 `thread_id`，Checkpointer 会自动隔离各自的对话状态。

### Q3: 系统重启后如何恢复对话？

**A**: 只要使用持久化的 Checkpointer（如 PostgreSQL），重启后使用相同的 `thread_id` 即可恢复到之前的对话状态。`InMemorySaver` 不支持重启恢复。

### Q4: 如何处理用户突然切换话题？

**A**: 在状态中维护 `current_topic` 字段，通过节点检测话题变化。检测到切换时可以：清空相关上下文、保存旧话题信息、初始化新话题上下文。

## 本节总结

对话管理与状态追踪是智能客服的核心能力。核心要点：

1. **MessagesState** 自动管理消息历史，无需手动维护上下文
2. **Checkpointer** 实现状态持久化，支持会话恢复
3. **SummarizationNode** 处理长对话压缩，控制 Token 消耗
4. **自定义状态字段** 追踪话题、实体、情感等业务信息

良好的对话管理让客服系统能够像人一样"记住"对话内容，提供连贯、自然的服务体验。
