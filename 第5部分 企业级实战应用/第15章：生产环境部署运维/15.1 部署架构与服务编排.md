# 15.1 部署架构与服务编排

## 概念讲解

将 LangChain 应用从开发环境推向生产环境，需要考虑服务如何拆分、如何部署、如何扩展等问题。LangGraph 应用由于其状态管理和长运行特性，在部署架构上有独特的考量。

### LangGraph 应用的部署特点

与传统 Web 应用不同，LangGraph 应用有以下特点：

1. **有状态执行**：通过 Checkpointer 维护对话状态，需要持久化存储支持
2. **长运行任务**：Agent 执行可能涉及多步工具调用，耗时较长
3. **流式输出**：支持实时响应，需要 WebSocket 或 SSE 连接
4. **资源密集**：LLM 调用和向量检索消耗大量计算和网络资源

## 核心要点

### 1. 无状态服务设计

LangGraph 应用的服务本身应该是无状态的。状态通过 Checkpointer 持久化到外部存储，这样服务实例可以随时扩展或重启而不丢失数据。

| 组件 | 部署方式 | 扩展策略 |
|------|----------|----------|
| LangGraph 服务 | 无状态，多副本 | 根据 CPU/内存使用率自动扩展 |
| PostgreSQL | 主从复制 | 读扩展用只读副本 |
| Redis | 集群模式 | 根据缓存命中率扩展 |
| 向量数据库 | 独立部署 | 根据查询 QPS 扩展 |

### 2. 流式响应支持

生产环境中应优先使用流式输出，提升用户体验：

- **SSE（Server-Sent Events）**：适合单向流式输出
- **WebSocket**：适合双向实时交互
- **HTTP 分块传输**：简单场景下的流式方案

### 3. 并发与限流

LLM API 通常有速率限制，需要在应用层实现：

- **令牌桶算法**：控制每秒请求数
- **并发连接池**：复用数据库和网络连接
- **请求队列**：高峰期排队处理，避免雪崩

## 简单示例

### FastAPI 部署 LangGraph 应用

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langgraph.graph import StateGraph, MessagesState, START
from langgraph.checkpoint.memory import InMemorySaver
from langchain.chat_models import init_chat_model
import json
import asyncio

app = FastAPI(title="LangGraph Agent API")

# 初始化模型和图（应用启动时）
model = init_chat_model("gpt-4o-mini")

def chatbot(state: MessagesState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

builder = StateGraph(MessagesState)
builder.add_node("chatbot", chatbot)
builder.add_edge(START, "chatbot")

checkpointer = InMemorySaver()  # 生产环境用 PostgresSaver
graph = builder.compile(checkpointer=checkpointer)

# 普通请求
@app.post("/chat")
async def chat(message: str, thread_id: str = "default"):
    config = {"configurable": {"thread_id": thread_id}}
    result = await graph.ainvoke(
        {"messages": [{"role": "user", "content": message}]},
        config
    )
    return {"response": result["messages"][-1].content}

# 流式请求
@app.post("/chat/stream")
async def chat_stream(message: str, thread_id: str = "default"):
    config = {"configurable": {"thread_id": thread_id}}
    
    async def event_generator():
        async for event in graph.astream(
            {"messages": [{"role": "user", "content": message}]},
            config,
            stream_mode="messages"
        ):
            chunk = event[0]
            if chunk.content:
                yield f"data: {json.dumps({'content': chunk.content}, ensure_ascii=False)}\n\n"
        yield "data: [DONE]\n\n"
    
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

### Docker 部署配置

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# 安装依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制代码
COPY . .

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# 启动服务
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]
```

## 进阶应用

### 1. Kubernetes 部署

在 K8s 环境中，使用 HorizontalPodAutoscaler 实现自动扩展：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: langgraph-agent
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: agent
        image: myregistry/langgraph-agent:latest
        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"
          limits:
            cpu: "2000m"
            memory: "2Gi"
        env:
        - name: LANGSMITH_TRACING
          value: "true"
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: langgraph-agent-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: langgraph-agent
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

## 常见问题

### Q1: 如何保证 LangGraph 服务的高可用？

**A**: 服务本身无状态，通过多副本部署 + 负载均衡实现。状态通过外部 Checkpointer（如 PostgreSQL）持久化，数据库使用主从复制保证可用性。

### Q2: 如何处理 LLM API 的速率限制？

**A**: 在应用层实现令牌桶限流，超出限制时排队等待或返回降级响应。同时可以使用多个 API Key 轮询分散请求。

### Q3: 流式响应中断如何处理？

**A**: 客户端实现重连机制，服务端通过 Checkpointer 保存中间状态，重连后从断点继续。

## 本节总结

LangGraph 应用的部署核心要点：

1. **无状态服务**：应用实例无状态，状态外置到 Checkpointer
2. **流式支持**：使用 SSE 或 WebSocket 提供实时响应
3. **弹性扩展**：根据负载自动伸缩服务实例
4. **容器化部署**：使用 Docker + K8s 实现标准化部署

合理的部署架构是 LangGraph 应用稳定运行的基础保障。
