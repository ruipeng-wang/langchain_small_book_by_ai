# 13.3.4 性能基准测试

## 概念讲解

**性能基准测试**是评估多代理系统在不同条件下的表现，帮助开发者识别瓶颈、优化策略和确保系统可扩展性的关键实践。对于基于 LangGraph 的多代理系统，性能测试涉及多个维度：执行时间、内存消耗、并发能力和状态管理开销。

### 性能测试的关键指标

| 指标 | 说明 | 测量方法 |
|------|------|---------|
| 延迟（Latency） | 从输入到输出的总时间 | `time.time()` 或 `time.perf_counter()` |
| 吞吐量（Throughput） | 单位时间内处理的任务数 | 任务数 / 总时间 |
| 内存使用 | 执行过程中的内存消耗 | `tracemalloc` 模块 |
| 状态开销 | 检查点存储的额外开销 | 比较有/无检查点的执行时间 |

### 性能优化的常见方向

1. **并行化**：将可并行的节点改为并行执行
2. **减少状态传输**：只传递必要的状态字段
3. **缓存结果**：避免重复计算相同输入
4. **异步 I/O**：利用异步操作提升 I/O 密集型任务性能

## 核心要点

- **基准测试需要对比场景**：单代理 vs 多代理、串行 vs 并行
- **关注实际场景而非极端值**：测试应反映真实使用模式
- **多次运行取平均值**：消除偶然因素影响
- **LangSmith 提供可视化性能分析**：追踪每个节点的延迟
- **检查点机制有性能开销**：权衡持久化需求和性能要求

## 简单示例

### 基础性能测试

```python
import time
from langgraph.graph import StateGraph, START, END
from typing import TypedDict
import statistics

class PerfState(TypedDict):
    data: str
    result: str

def slow_node(state: PerfState):
    """模拟耗时操作"""
    time.sleep(0.1)  # 模拟 100ms 的 I/O 操作
    return {"result": f"处理完成: {state['data']}"}

def fast_node(state: PerfState):
    """快速操作"""
    return {"result": f"快速处理: {state['data']}"}

def benchmark(graph, state: dict, runs: int = 10) -> dict:
    """性能测试函数"""
    times = []
    for _ in range(runs):
        start = time.perf_counter()
        graph.invoke(state)
        end = time.perf_counter()
        times.append(end - start)
    
    return {
        "avg": statistics.mean(times),
        "min": min(times),
        "max": max(times),
        "p95": sorted(times)[int(len(times) * 0.95)]
    }

# 测试单节点性能
builder = StateGraph(PerfState)
builder.add_node("process", slow_node)
builder.add_edge(START, "process")
builder.add_edge("process", END)
graph = builder.compile()

results = benchmark(graph, {"data": "test", "result": ""})
print(f"平均延迟: {results['avg']:.3f}s")
print(f"P95延迟: {results['p95']:.3f}s")
```

### 串行 vs 并行对比测试

```python
# 串行工作流
serial_builder = StateGraph(PerfState)
serial_builder.add_node("step1", slow_node)
serial_builder.add_node("step2", slow_node)
serial_builder.add_node("step3", slow_node)
serial_builder.add_edge(START, "step1")
serial_builder.add_edge("step1", "step2")
serial_builder.add_edge("step2", "step3")
serial_builder.add_edge("step3", END)
serial_graph = serial_builder.compile()

# 并行工作流
parallel_builder = StateGraph(PerfState)
parallel_builder.add_node("step1", slow_node)
parallel_builder.add_node("step2", slow_node)
parallel_builder.add_node("step3", slow_node)
parallel_builder.add_edge(START, "step1")
parallel_builder.add_edge(START, "step2")
parallel_builder.add_edge(START, "step3")
parallel_builder.add_edge("step1", END)
parallel_builder.add_edge("step2", END)
parallel_builder.add_edge("step3", END)
parallel_graph = parallel_builder.compile()

# 对比测试
print("串行执行:")
serial_results = benchmark(serial_graph, {"data": "test", "result": ""})
print(f"  平均: {serial_results['avg']:.3f}s")

print("并行执行:")
parallel_results = benchmark(parallel_graph, {"data": "test", "result": ""})
print(f"  平均: {parallel_results['avg']:.3f}s")
print(f"  加速比: {serial_results['avg']/parallel_results['avg']:.1f}x")
```

## 进阶应用

### 内存使用分析

```python
import tracemalloc

def memory_test():
    """测试内存使用"""
    tracemalloc.start()
    
    # 执行工作流
    for _ in range(100):
        graph.invoke({"data": "test", "result": ""})
    
    current, peak = tracemalloc.get_traced_memory()
    tracemalloc.stop()
    
    print(f"当前内存: {current / 1024:.1f} KB")
    print(f"峰值内存: {peak / 1024:.1f} KB")

memory_test()
```

### 检查点开销测试

```python
from langgraph.checkpoint.memory import MemorySaver

# 无检查点版本
graph_no_checkpoint = builder.compile()

# 有检查点版本
graph_with_checkpoint = builder.compile(checkpointer=MemorySaver())
config = {"configurable": {"thread_id": "perf-test"}}

def invoke_with_config(graph, state, config=None):
    if config:
        return graph.invoke(state, config)
    return graph.invoke(state)

# 对比测试
print("无检查点:")
no_cp_results = benchmark(
    lambda s: invoke_with_config(graph_no_checkpoint, s), 
    {"data": "test", "result": ""}
)
print(f"  平均: {no_cp_results['avg']:.3f}s")

print("有检查点:")
with_cp_results = benchmark(
    lambda s: invoke_with_config(graph_with_checkpoint, s, config),
    {"data": "test", "result": ""}
)
print(f"  平均: {with_cp_results['avg']:.3f}s")
print(f"  开销: {(with_cp_results['avg']/no_cp_results['avg']-1)*100:.1f}%")
```

## 常见问题

### Q1: 为什么并行执行没有显著提升？

**A**: 可能的原因包括：1）节点执行时间太短，并行开销抵消了收益；2）受限于全局解释器锁（GIL）；3）I/O 操作已异步化。建议先分析瓶颈再决定优化方向。

### Q2: 如何测试大状态下的性能？

**A**: 增加状态数据量（如大型消息列表），测试状态复制和合并的开销。

### Q3: LangSmith 能帮助性能分析吗？

**A**: 可以。LangSmith 追踪每个节点的执行时间，帮助识别慢节点。

### Q4: 如何设置合理的性能目标？

**A**: 根据业务需求设定。交互式应用通常要求 P95 延迟 < 2 秒，批处理应用关注吞吐量。

## 本节总结

性能基准测试是多代理系统开发和优化的重要环节：

1. 建立系统的性能基线，识别瓶颈
2. 对比不同架构方案的性能表现
3. 分析内存使用模式，优化状态管理
4. 评估检查点等功能的性能开销

定期进行性能测试，确保系统在生产环境中的稳定性。
