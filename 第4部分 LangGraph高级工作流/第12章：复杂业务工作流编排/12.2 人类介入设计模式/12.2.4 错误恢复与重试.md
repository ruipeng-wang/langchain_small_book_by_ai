# 12.2.4 错误恢复与重试

## 概念讲解

在生产环境中，工作流执行过程中不可避免地会遇到各种错误：网络超时、API 限流、模型调用失败等。**错误恢复与重试**机制确保工作流能够在遇到暂时性错误时自动恢复，而不是直接失败。

LangGraph 的错误恢复策略分为两个层面：

1. **节点级别重试**：在单个节点内部实现重试逻辑
2. **工作流级别恢复**：利用检查点机制，从失败的节点恢复整个工作流

### 错误分类

| 错误类型 | 特点 | 处理策略 |
|---------|------|---------|
| 暂时性错误 | 网络超时、API 限流 | 自动重试 |
| 永久性错误 | 参数错误、认证失败 | 跳过或降级 |
| 业务错误 | 内容审核不通过 | 人工介入 |

### 重试策略

- **指数退避**：每次重试等待时间指数增长，避免频繁重试
- **最大重试次数**：限制重试上限，防止无限循环
- **退避策略**：重试失败后执行降级逻辑

## 核心要点

- **节点内部实现重试逻辑**：LangGraph 不内置重试，由节点函数自行处理
- **检查点支持失败恢复**：失败前的状态已保存，可从检查点恢复
- **更新状态实现重试**：通过 `update_state` 修改状态后重新执行
- **降级模式处理永久失败**：重试上限后的备选方案
- **人工介入处理复杂错误**：利用 HITL 模式请求人工决策

## 简单示例

### 节点内部重试

最常见的重试模式是在节点函数内部使用 `tenacity` 库：

```python
from tenacity import retry, stop_after_attempt, wait_exponential
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class State(TypedDict):
    query: str
    result: str
    retry_count: int

# 使用装饰器定义重试
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10)
)
def call_llm_with_retry(query: str) -> str:
    """模拟可能失败的 LLM 调用"""
    # 实际项目中调用真实的 LLM API
    return f"查询结果: {query}"

def process_node(state: State):
    """带重试的处理节点"""
    try:
        result = call_llm_with_retry(state["query"])
        return {"result": result, "retry_count": 0}
    except Exception as e:
        return {"result": f"处理失败: {str(e)}", "retry_count": 3}

builder = StateGraph(State)
builder.add_node("process", process_node)
builder.add_edge(START, "process")
builder.add_edge("process", END)

graph = builder.compile()
result = graph.invoke({"query": "测试查询", "result": "", "retry_count": 0})
```

### 手动重试逻辑

不依赖外部库的手动重试实现：

```python
import time

def retry_with_backoff(func, max_retries=3, base_delay=1):
    """指数退避重试"""
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries - 1:
                raise  # 最后一次重试，抛出异常
            delay = base_delay * (2 ** attempt)
            print(f"第 {attempt + 1} 次重试，等待 {delay} 秒")
            time.sleep(delay)

def robust_llm_node(state: State):
    """健壮性更强的 LLM 节点"""
    def llm_call():
        # 模拟 LLM 调用
        return f"处理: {state['query']}"
    
    try:
        result = retry_with_backoff(llm_call)
        return {"result": result}
    except Exception:
        return {"result": "重试失败，使用默认响应"}
```

## 进阶应用

### 检查点恢复模式

利用检查点从失败的节点恢复：

```python
from langgraph.checkpoint.memory import MemorySaver

# 配置检查点器
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)
config = {"configurable": {"thread_id": "recovery-demo"}}

# 首次执行可能失败
try:
    result = graph.invoke({"query": "测试", "result": "", "retry_count": 0}, config)
except Exception as e:
    print(f"执行失败: {e}")
    
    # 检查当前状态
    state = graph.get_state(config)
    print(f"失败时的状态: {state.values}")
    print(f"下一个要执行的节点: {state.next}")
    
    # 修改状态后重新执行
    graph.update_state(config, {"retry_count": state.values["retry_count"] + 1})
    result = graph.invoke(None, config)  # 从检查点恢复继续执行
```

### 降级策略

当重试失败时的降级处理：

```python
def fallback_node(state: State):
    """降级节点，在主节点失败后执行"""
    return {"result": f"[降级] 无法处理查询: {state['query']}"}

def should_fallback(state: State) -> str:
    """决定是否需要降级"""
    if state["result"].startswith("处理失败"):
        return "fallback"
    return "success"

builder = StateGraph(State)
builder.add_node("process", process_node)
builder.add_node("fallback", fallback_node)

builder.add_edge(START, "process")
builder.add_conditional_edges("process", should_fallback, {
    "fallback": "fallback",
    "success": END
})
builder.add_edge("fallback", END)
```

## 常见问题

### Q1: LangGraph 是否有内置的重试机制？

**A**: LangGraph 本身不内置重试机制，推荐在节点函数内部实现重试逻辑。可以使用 `tenacity` 等成熟的重试库。

### Q2: 节点失败后如何从检查点恢复？

**A**: 配置检查点器后，节点失败前的状态会自动保存。通过 `get_state` 查看当前状态，使用 `update_state` 修改状态后，再次调用 `invoke(None, config)` 即可从检查点恢复。

### Q3: 如何在重试时保留部分状态？

**A**: 在节点返回值中包含需要保留的状态字段。使用 `update_state` 可以精确控制要更新的状态字段。

### Q4: 并行节点中一个失败如何处理？

**A**: 默认情况下，并行节点中一个失败会导致整个工作流失败。可以在每个并行节点内部捕获异常并返回降级结果，或者使用归约器处理错误状态。

## 本节总结

错误恢复与重试是生产环境工作流的必备能力。LangGraph 通过灵活的节点设计和检查点机制，支持多种错误恢复策略：

1. 节点内部使用 `tenacity` 等库实现自动重试
2. 检查点机制支持从失败节点恢复
3. 降级策略处理重试失败后的备选方案
4. 人工介入处理复杂的错误决策

合理的错误恢复策略能显著提升工作流的稳定性和用户体验。
