# 12.1.1 任务分解与依赖分析

在复杂业务工作流中，任务分解与依赖分析是构建健壮系统的关键基础。LangGraph通过其图计算引擎提供了强大的能力来建模任务间的依赖关系，确保工作流按预期顺序执行。

## 概念讲解

### 任务分解的核心思想
任务分解是指将复杂的业务过程拆分为多个原子操作，每个操作称为一个"节点"。依赖分析则是确定这些节点之间的执行顺序关系，形成有向无环图（DAG）结构。

### LangGraph的任务建模优势
LangGraph的StateGraph不仅支持简单的线性任务流，还能处理复杂的依赖关系：
1. **显式依赖声明**：通过`add_edge()`方法直接定义节点间的执行顺序
2. **条件依赖**：根据运行时状态动态决定下一个节点
3. **并行依赖**：多个节点可以同时满足依赖条件后执行后续节点
4. **循环依赖检测**：自动防止循环依赖导致的死锁

### 依赖类型分类
- **强依赖**：前序任务必须成功完成
- **弱依赖**：前序任务可以失败，但会尝试恢复
- **资源依赖**：共享资源访问需要同步控制
- **数据依赖**：任务间的数据传递关系

## 核心要点

### 1. 状态驱动的依赖管理
LangGraph使用共享状态对象来传递数据和控制流程，每个节点都能访问和修改全局状态，形成隐式的数据依赖。

### 2. 类型安全的依赖声明
通过TypedDict定义状态类型，编译器能在编译时检查依赖关系的合法性。

### 3. 动态依赖解析
依赖关系不仅可以在编译时定义，还能在运行时根据状态动态调整。

### 4. 依赖可视化
编译后的图结构可以导出为可视化格式，便于调试和优化。

## 简单示例

以下是一个电商订单处理的任务分解示例，展示了如何定义任务节点和依赖关系：

```python
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
import operator

# 定义订单处理状态
class OrderState(TypedDict):
    order_id: str
    customer_info: dict
    payment_status: Literal["pending", "paid", "failed"]
    inventory_status: dict
    shipping_info: dict
    processing_steps: Annotated[list[str], operator.add]
    errors: Annotated[list[str], operator.add]

# 1. 订单验证节点
def validate_order(state: OrderState) -> dict:
    """验证订单基本信息"""
    steps = ["订单验证"]
    errors = []
    
    if not state.get("order_id"):
        errors.append("订单ID不能为空")
    if not state.get("customer_info"):
        errors.append("客户信息缺失")
    
    return {
        "processing_steps": steps,
        "errors": errors,
        "inventory_status": {"checked": len(errors) == 0}
    }

# 2. 库存检查节点
def check_inventory(state: OrderState) -> dict:
    """检查商品库存"""
    steps = ["库存检查"]
    errors = []
    
    # 模拟库存检查逻辑
    inventory_ok = state.get("inventory_status", {}).get("checked", False)
    if not inventory_ok:
        errors.append("库存检查失败")
    
    return {
        "processing_steps": steps,
        "errors": errors,
        "inventory_status": {"available": len(errors) == 0}
    }

# 3. 支付处理节点
def process_payment(state: OrderState) -> dict:
    """处理支付"""
    steps = ["支付处理"]
    errors = []
    
    # 模拟支付处理
    payment_success = True  # 模拟支付成功
    if payment_success:
        return {
            "processing_steps": steps,
            "payment_status": "paid"
        }
    else:
        errors.append("支付失败")
        return {
            "processing_steps": steps,
            "errors": errors,
            "payment_status": "failed"
        }

# 4. 发货准备节点
def prepare_shipping(state: OrderState) -> dict:
    """准备发货"""
    steps = ["发货准备"]
    
    # 只有支付成功且库存充足时才发货
    if state.get("payment_status") == "paid" and \
       state.get("inventory_status", {}).get("available", False):
        return {
            "processing_steps": steps,
            "shipping_info": {
                "status": "ready",
                "tracking_number": "TRK123456789"
            }
        }
    return {
        "processing_steps": steps,
        "errors": ["发货条件不满足"]
    }

# 构建任务依赖图
def build_order_processing_graph():
    builder = StateGraph(OrderState)
    
    # 添加任务节点
    builder.add_node("validate", validate_order)
    builder.add_node("check_inventory", check_inventory)
    builder.add_node("process_payment", process_payment)
    builder.add_node("prepare_shipping", prepare_shipping)
    
    # 定义任务依赖关系
    builder.add_edge(START, "validate")
    builder.add_edge("validate", "check_inventory")
    builder.add_edge("check_inventory", "process_payment")
    
    # 条件依赖：只有支付成功才执行发货
    def should_ship(state: OrderState) -> Literal["prepare_shipping", END]:
        if state.get("payment_status") == "paid" and \
           state.get("inventory_status", {}).get("available", False):
            return "prepare_shipping"
        return END
    
    builder.add_conditional_edges("process_payment", should_ship)
    builder.add_edge("prepare_shipping", END)
    
    # 编译带检查点的图
    return builder.compile(checkpointer=InMemorySaver())

# 执行工作流
graph = build_order_processing_graph()
initial_state = {
    "order_id": "ORD123456",
    "customer_info": {"name": "张三", "email": "zhangsan@example.com"},
    "payment_status": "pending",
    "inventory_status": {},
    "shipping_info": {},
    "processing_steps": [],
    "errors": []
}

config = {"configurable": {"thread_id": "order-123"}}
result = graph.invoke(initial_state, config)
print(f"处理步骤: {result['processing_steps']}")
print(f"支付状态: {result['payment_status']}")
print(f"发货信息: {result.get('shipping_info', {})}")
```

## 常见问题

### Q1: 如何调试复杂的依赖关系？
**A**: 使用`graph.get_graph().draw_mermaid()`生成依赖图可视化，或者通过`graph.get_graph().to_json()`导出图结构进行分析。

### Q2: 任务间的数据依赖如何处理？
**A**: LangGraph通过共享状态对象自动管理数据依赖，每个节点都可以读取和修改状态，后续节点会看到前序节点的状态变更。

### Q3: 如何处理动态变化的依赖关系？
**A**: 使用条件边（`add_conditional_edges`）或`Command`类型在运行时动态决定下一个节点，或者使用`Send`API创建并行执行路径。

### Q4: 依赖分析对性能有什么影响？
**A**: LangGraph在编译时会优化依赖关系，生成高效的执行计划。复杂的依赖分析主要在编译时完成，运行时开销很小。

## 本节总结

任务分解与依赖分析是构建复杂业务工作流的核心技术。LangGraph通过以下特性提供了强大的支持：

1. **声明式依赖定义**：通过简洁的API定义任务节点和依赖关系
2. **类型安全**：利用Python类型提示确保依赖关系的正确性
3. **动态灵活性**：支持运行时调整依赖关系和任务结构
4. **可视化调试**：提供丰富的工具来可视化和调试依赖关系
5. **性能优化**：编译时优化确保复杂的依赖分析不影响运行时性能

掌握这些技术后，您可以构建出能够处理各种复杂业务场景的健壮工作流系统。在实际应用中，建议先从简单的线性依赖开始，逐步添加条件分支和并行执行，最后处理复杂的循环和动态依赖。
