# 12.1.4 并行执行优化

## 概念讲解

在实际的业务场景中，很多工作流包含多个可以**同时执行**的独立任务。如果按顺序执行这些任务，会导致不必要的等待时间。LangGraph 提供了原生的**并行执行**支持，通过**扇出（Fan-out）**和**扇入（Fan-in）**模式，可以显著提升工作流的执行效率。

### 并行执行的工作原理

LangGraph 的并行执行基于**多目标边**实现：当一个节点通过边连接到多个后续节点时，这些后续节点会并行执行。执行完成后，多个节点的状态更新会通过**归约器（Reducer）**合并到共享状态中。

```
         ┌─── 节点A ───┐
         │             │
起始节点 ─┼─── 节点B ───┼─→ 聚合节点
         │             │
         └─── 节点C ───┘
```

### 并行执行的两种模式

1. **Graph API 模式**：通过 `add_edge` 连接多个目标，适合结构化工作流
2. **Functional API 模式**：通过 `@task` 装饰器和 `Future` 对象，适合动态并行任务

### 归约器（Reducer）

当多个并行节点同时更新状态时，需要定义如何合并这些更新。LangGraph 使用 `Annotated` 类型配合归约函数来处理并发状态更新。

## 核心要点

- **多目标边触发并行执行**：从同一节点出发的多条边会并行执行目标节点
- **归约器处理并发状态更新**：使用 `operator.add` 等函数合并多个节点的结果
- **Functional API 支持动态并行**：运行时决定并行任务数量
- **Send API 支持动态扇出**：根据数据动态创建并行任务
- **并行执行是 I/O 密集型优化**：特别适合 LLM API 调用等耗时操作

## 简单示例

### 基础并行执行

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    topic: str
    joke: str
    story: str
    combined_output: str

# 并行节点
def generate_joke(state: State):
    """生成笑话"""
    return {"joke": f"关于 {state['topic']} 的笑话"}

def generate_story(state: State):
    """生成故事"""
    return {"story": f"关于 {state['topic']} 的故事"}

# 聚合节点
def aggregate(state: State):
    """合并并行节点的结果"""
    combined = f"主题：{state['topic']}\n"
    combined += f"笑话：{state['joke']}\n"
    combined += f"故事：{state['story']}"
    return {"combined_output": combined}

# 构建并行工作流
builder = StateGraph(State)
builder.add_node("joke", generate_joke)
builder.add_node("story", generate_story)
builder.add_node("aggregate", aggregate)

# 关键：从 START 连接到多个节点，实现并行执行
builder.add_edge(START, "joke")
builder.add_edge(START, "story")
# 并行节点完成后，都流向聚合节点
builder.add_edge("joke", "aggregate")
builder.add_edge("story", "aggregate")
builder.add_edge("aggregate", END)

graph = builder.compile()
result = graph.invoke({"topic": "人工智能", "joke": "", "story": "", "combined_output": ""})
print(result["combined_output"])
```

### 使用归约器合并列表结果

```python
class ReduceState(TypedDict):
    items: Annotated[list, operator.add]  # 使用 add 归约器合并列表

def task_a(state: ReduceState):
    return {"items": ["结果A"]}

def task_b(state: ReduceState):
    return {"items": ["结果B"]}

builder = StateGraph(ReduceState)
builder.add_node("task_a", task_a)
builder.add_node("task_b", task_b)
builder.add_edge(START, "task_a")
builder.add_edge(START, "task_b")
builder.add_edge("task_a", END)
builder.add_edge("task_b", END)

graph = builder.compile()
result = graph.invoke({"items": []})
print(result["items"])  # 输出: ['结果A', '结果B']
```

## 进阶应用

### Send API 动态扇出

当并行任务数量需要根据输入数据动态决定时，使用 `Send` API：

```python
from langgraph.types import Send

class MapReduceState(TypedDict):
    topics: list[str]
    results: Annotated[list, operator.add]
    summary: str

def generate_topics(state: MapReduceState):
    """生成待处理的主题列表"""
    return {"topics": ["机器学习", "深度学习", "强化学习"]}

def process_topic(state: MapReduceState):
    """处理单个主题（会被多次调用）"""
    topic = state["subject"]  # Send 传递的参数
    return {"results": [f"处理完成: {topic}"]}

def summarize(state: MapReduceState):
    """汇总所有结果"""
    return {"summary": f"共处理 {len(state['results'])} 个主题"}

# 动态扇出函数
def fan_out(state: MapReduceState):
    """根据 topics 动态创建 Send 任务"""
    return [Send("process_topic", {"subject": t}) for t in state["topics"]]

builder = StateGraph(MapReduceState)
builder.add_node("generate_topics", generate_topics)
builder.add_node("process_topic", process_topic)
builder.add_node("summarize", summarize)

builder.add_edge(START, "generate_topics")
# 条件边实现动态扇出
builder.add_conditional_edges("generate_topics", fan_out, ["process_topic"])
builder.add_edge("process_topic", "summarize")
builder.add_edge("summarize", END)

graph = builder.compile()
```

### Functional API 并行

使用装饰器风格的 Functional API 实现并行：

```python
from langgraph.func import entrypoint, task

@task
def process_topic(topic: str) -> str:
    """处理单个主题的任务"""
    # 模拟 LLM 调用等 I/O 操作
    return f"处理完成: {topic}"

@entrypoint()
def parallel_workflow(topics: list[str]) -> str:
    """并行处理多个主题"""
    # 创建并行任务
    futures = [process_topic(topic) for topic in topics]
    # 等待所有结果
    results = [f.result() for f in futures]
    return f"共完成 {len(results)} 个任务"

# 执行
result = parallel_workflow.invoke(["主题A", "主题B", "主题C"])
```

## 常见问题

### Q1: 并行执行是真正的多线程吗？

**A**: LangGraph 的并行执行主要是异步并发，不是多线程。对于 I/O 密集型任务（如 LLM 调用），异步并发已经能提供显著的性能提升。

### Q2: 并行节点执行顺序如何保证？

**A**: 并行节点的执行顺序不保证，但最终的状态合并是确定性的。如果需要特定顺序，应该使用串行边而非并行边。

### Q3: 如何处理并行节点的错误？

**A**: 默认情况下，一个并行节点失败会导致整个工作流失败。可以通过在节点内部添加错误处理逻辑来实现容错。

### Q4: Send API 和普通并行边有什么区别？

**A**: 普通并行边在编译时确定目标节点；Send API 可以在运行时动态决定目标节点和参数，更适合 Map-Reduce 模式。

## 本节总结

并行执行优化是提升 LangGraph 工作流性能的关键手段。通过多目标边、归约器和 Send API，可以实现从简单并行到动态 Map-Reduce 的多种并行模式。核心要点包括：

1. 多目标边触发并行执行，提升 I/O 密集型任务性能
2. 归约器处理并发状态更新，避免数据冲突
3. Send API 支持运行时动态扇出
4. Functional API 提供更简洁的并行编程模型

合理利用并行执行，可以显著减少工作流的总体执行时间。
