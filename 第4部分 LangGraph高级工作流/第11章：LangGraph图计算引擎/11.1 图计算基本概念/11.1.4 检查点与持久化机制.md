# 11.1.4 检查点与持久化机制

## 概念讲解

在构建复杂的工作流时，**检查点（Checkpoint）**和**持久化（Persistence）**是两个至关重要的概念。检查点机制允许工作流在执行过程中的任意时刻保存当前状态，而持久化则确保这些状态能够被长期存储并在需要时恢复。

LangGraph 通过**检查点器（Checkpointer）**抽象提供了开箱即用的持久化能力。每次节点执行完成后，检查点器会自动保存状态快照。这使得工作流具备了**持久执行（Durable Execution）**的能力——即使进程重启或系统崩溃，工作流也能从上次检查点处恢复执行。

### 为什么需要检查点？

考虑以下场景：

1. **长时间运行的工作流**：一个审批流程可能需要等待人工审批数小时甚至数天
2. **错误恢复**：当某个节点执行失败时，能够从失败点恢复而非重新开始
3. **人工介入（Human-in-the-loop）**：在需要人工审核时暂停执行，等待人类决策后继续
4. **时间旅行调试**：查看工作流历史状态，理解执行路径

### 检查点器类型

LangGraph 支持多种检查点器实现：

- **InMemorySaver**：内存存储，适用于开发和测试
- **SqliteSaver**：SQLite 数据库存储，适用于轻量级生产环境
- **PostgresSaver**：PostgreSQL 存储，适用于高可用生产环境
- **自定义检查点器**：通过实现 `BaseCheckpointSaver` 接口，可以对接任意存储后端

## 核心要点

- **检查点是状态的时间快照**：记录工作流在某一时刻的完整状态
- **Thread ID 是状态的主键**：通过 `thread_id` 区分不同会话的状态
- **检查点器自动管理生命周期**：节点执行前后自动保存和恢复状态
- **支持历史状态查询**：可以回溯查看任意历史检查点
- **子图检查点自动传播**：父图的检查点器会传递给子图

## 简单示例

### 基础检查点配置

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from typing import TypedDict

class State(TypedDict):
    messages: list
    step_result: str

def step_one(state: State):
    print("执行步骤一")
    return {"step_result": "步骤一完成"}

def step_two(state: State):
    print("执行步骤二")
    return {"step_result": "步骤二完成"}

# 构建图
builder = StateGraph(State)
builder.add_node("step_one", step_one)
builder.add_node("step_two", step_two)
builder.add_edge(START, "step_one")
builder.add_edge("step_one", "step_two")
builder.add_edge("step_two", END)

# 配置检查点器
checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 执行时必须指定 thread_id
config = {"configurable": {"thread_id": "session-001"}}
result = graph.invoke({"messages": [], "step_result": ""}, config)
print(result)
```

### 查看状态历史

```python
# 获取状态历史
history = list(graph.get_state_history(config))

for state in history:
    print(f"步骤: {state.metadata.get('step')}")
    print(f"状态: {state.values}")
    print(f"下一个节点: {state.next}")
    print("---")
```

## 进阶应用

### 使用 SQLite 持久化

对于需要长期存储的场景，SQLite 是比内存更合适的选择：

```python
import sqlite3
from langgraph.checkpoint.sqlite import SqliteSaver

# 创建 SQLite 连接
conn = sqlite3.connect("checkpoints.db", check_same_thread=False)
checkpointer = SqliteSaver(conn)

# 编译图时指定检查点器
graph = builder.compile(checkpointer=checkpointer)

# 多次执行，状态会持久化到数据库
for i in range(3):
    config = {"configurable": {"thread_id": f"session-{i}"}}
    graph.invoke({"messages": [], "step_result": ""}, config)
```

### 状态恢复与修改

有时需要回退到历史状态并修改后重新执行：

```python
# 获取最新状态
current_state = graph.get_state(config)
print(f"当前状态: {current_state.values}")

# 修改状态并重新执行
graph.update_state(config, {"step_result": "手动修改的结果"})
result = graph.invoke(None, config)  # 从当前状态继续
```

### 过滤历史检查点

```python
history = list(graph.get_state_history(config))

# 找到在某个节点执行前的检查点
before_node_b = next(s for s in history if s.next == ("step_two",))

# 按步骤号过滤
step_2 = next(s for s in history if s.metadata["step"] == 2)
```

## 常见问题

### Q1: 什么时候会创建检查点？

**A**: 每次节点执行完成后，LangGraph 会自动创建一个检查点。检查点包含了节点的执行结果和最新的状态。

### Q2: thread_id 的作用是什么？

**A**: `thread_id` 是检查点的主键，用于区分不同会话的状态。相同 `thread_id` 的多次调用会共享状态历史，不同 `thread_id` 则完全隔离。

### Q3: 内存检查点和数据库检查点有什么区别？

**A**: 内存检查点（`InMemorySaver`）在进程重启后会丢失所有数据，适合开发测试。数据库检查点（`SqliteSaver`、`PostgresSaver`）会持久化到磁盘，支持进程重启后恢复。

### Q4: 如何清理过期的检查点？

**A**: 对于 SQLite，可以直接执行 SQL 删除；对于 Postgres，可以使用内置的清理函数。建议定期清理不再需要的历史检查点以节省存储空间。

## 本节总结

检查点与持久化机制是 LangGraph 工作流可靠运行的基石。通过检查点器抽象，开发者可以灵活选择存储后端，从内存到数据库，甚至自定义实现。核心要点包括：

1. 检查点是状态的时间快照，支持持久执行和恢复
2. Thread ID 是状态的主键，区分不同会话
3. LangGraph 提供多种检查点器实现
4. 支持状态历史查询和时间旅行调试
5. 子图检查点会自动从父图继承
